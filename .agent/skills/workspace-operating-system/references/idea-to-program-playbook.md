# Idea To Program Playbook

Use this playbook when the input is an idea, a vision, or a vague ambitious request rather than a ready-to-build specification.

## Goal

Convert raw ambition into an executable engineering program without pretending the whole thing is already well-defined.

## Step 1: Name The Outcome

Write the problem in concrete terms:

- who benefits
- what changes for them
- what success looks like
- what failure would look like

If the user gives a fantasy-level concept, keep the ambition but translate it into a software and systems outcome.

## Step 2: Split Vision From First Delivery

Always separate:

- long-term vision
- first useful milestone
- first verifiable slice

The first slice should prove the direction, not fulfill the whole dream.

## Step 3: Build The Program Brief

Every serious idea should be turned into a short brief:

- mission statement
- target users or operators
- primary jobs to be done
- constraints
- known unknowns
- out-of-scope items

## Step 4: Capture Requirement Layers

Do not stop at features. Capture the stack of requirements:

- functional requirements
- non-functional requirements
- operational requirements
- safety or compliance constraints
- observability requirements
- handoff and support expectations

## Step 5: Define Verification Early

Before major implementation, decide how claims will be proven:

- mock or prototype
- local simulation
- automated tests
- browser or runtime checks
- hardware-in-the-loop or lab validation
- staged environment verification

If no proof path exists, the idea is still a concept, not a deliverable.

## Step 6: Slice The Program

Use slices that reduce uncertainty:

1. intake and framing
2. architecture and interfaces
3. prototype or simulation
4. first integrated vertical slice
5. operational hardening
6. rollout and support

## Step 7: Assign Lanes

Map the work to lanes:

- product
- architecture
- implementation
- QA
- security
- release and ops
- docs and enablement

## Artifact Set

A strong idea-to-program conversion usually produces:

- brief or PRD
- architecture decision set
- milestone plan
- risk register
- acceptance criteria
- verification plan
- next branch or execution step

## Red Flags

Slow down if:

- the idea needs physics, hardware, regulation, or budget assumptions that are still unstated
- the first slice is too large to verify
- requirements are written as vibes instead of outcomes
- "done" means "everything" instead of a real milestone
